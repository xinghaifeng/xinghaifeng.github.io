  • AWS EC2虚机介绍

AWS EC2是亚马逊云主打产品，也是用户使用最广泛的产品，相当于传统机房的一台物理机器，可以配置多硬盘、多网卡等等

闲言碎语不多讲，直接上图创建一台EC2：
1、登陆到控制台，选择EC2服务

2、启动EC2实例，开始创建EC2

3、选择EC2实例操作系统，这里我们默认AWS提供的Linux

4、选择EC2机型配置，可以根据需求、价格等因素选择

5、配置详细信息，启动机器数量、网络、IAM、监控

6、配置存储，往EC2上挂硬盘

7、配置安全组策略，开放哪些端口，设置白名单

8、审查启动实例 ，仔细检查一遍配置

9、创建PEM密钥对 ，一定要保存好，以后登录全靠它，没有就煞笔了

10、分配静态IP地址，如果创建EC2时候没有分配，那么实例创建后最好分配一下 ，否则重启实例IP会漂移的！

11、EC2创建完成后仍然可以管理安全组，设置白名单

12、使用PUTTY登录配置，下载puttygen将pem文件生成ppk文件（1）

13、使用PUTTY登录配置（2）

14、连接后，输入ec2-user敲回车，无需密码，登陆啦！

15、更多机型供选择:https://aws.amazon.com/cn/ec2/instance-types/

  • AWS IAM用户角色权限

IAM全称： Identity and Access Management ，IAM能够集中管理用户、安全证书（例如访问密钥），以及控制用户可访问哪些AWS 资源的权限。可以按角色、权限、组来划分用户、可以将EC2绑定到角色中。公司可以通过IAM划分各个部门的权限，方便统一管理及账单整合费用计算。

账号登录方式：用户名+密码+MFA 设备/手机令牌 ，这样即使用户名密码暴露，账号也不会被恶意登陆，每个账号单独认证，互不影响 ，如果用户需要以API编程方式访问AWS，那么必须为他们创建访问密钥。

##AWS IAM

官方罗列的IAM 功能：
• 集中控制用户和安全证书– 您可控制每个用户的AWS 安全证书（例如访问密钥）的创建、交替及撤销
• 集中控制用户访问– 您可控制AWS 系统中用户可访问的数据及访问的方式
• 共享的AWS 资源– 用户可共享合作项目的数据
• 基于组织群组的许可– 您可基于用户的职责（例如，管理员、开发人员等）或部门来限制用户的访问。当用户调入组织内部时，您可轻松更新他们对AWS 的访问，以反映他们角色的变化
• 集中控制AWS 资源– 您的组织对用户创建的数据采取集中控制，这样当用户调入或离开组织时，可确保数据的连续性或不会遗失数据
• 控制资源的创建– 您可帮助确保用户仅在获得批准的位置创建AWS 数据
• 联网控制– 您可帮助确保用户仅使用SSL 从组织的公司网络访问AWS 资源
• 单一AWS 账单–您组织的AWS 账户将收到针对所有用户的AWS 活动而开具的单一AWS 账单

命令行设置权限：
arn:aws:service:region:account:resource
• service 标识AWS 产品。对于IAM 资源，它始终为iam。
• region 是资源所在的地区。对于IAM 资源，它始终留为空白。
• account 是指AWS 账户ID，不含连字符（例如，123456789012）
• resource 是指识别特定资源的部分

例如：
arn:aws:iam::123456789012:root
arn:aws:iam::123456789012:user/Bob
arn:aws:iam::123456789012:user/division_abc/subdivision_xyz/Bob
arn:aws:iam::123456789012:group/Developers
arn:aws:iam::123456789012:group/division_abc/subdivision_xyz/product_A/Develop
arn:aws:iam::123456789012:user/*

策略实例，都是JSON格式的字符：
{
“Version”: “2012-10-17”,
“Statement”: [{
“Effect”: “Allow”,
“Action”: [“iam:AccessKey“],
“Resource”: “arn:aws:iam::ACCOUNT-ID-WITHOUT-HYPHENS:user/division_abc/sub
division_xyz/Bob”
}]
}

{
“Version”: “2012-10-17”,
“Statement”: [
{
“Effect”: “Allow”,
“Action”: “s3:“,
“Resource”: “arn:aws:s3:::example_bucket/marketing/newproductlaunch/wid
get/“
},
{
“Effect”: “Allow”,
“Action”: “s3:ListBucket“,
“Resource”: “arn:aws:s3:::example_bucket”,
“Condition”: {“StringLike”: {“s3:prefix”: “marketing/newproductlaunch/wid
get/“}}
}
]
}

当IAM 创建用户、群组、实例配置文件或服务器证书时，它会向每个实体分配唯一的ID，如以下示例所示：AIDAJQABLZS4A3QDU576Q在大多数情况下，当您使用IAM 实体时，将使用易记名称和ARN，因此不需要知道特定实体的唯一ID。然而，当无法使用易记名称时，唯一ID 有时也是有用的。

总结：
1、可以使用AWS CloudTrail 记录IAM 事件
2、在IAM管理时候建议用命令行脚本方式配置。
3、尽量不要用超级管理员权限，登陆EC2后尽量不要用root,再划分管理员权限

再引用知乎上凡尘-高级百度产品经理的一段解析：

百度国际hao123产品团队正好是我负责的，楼上也有贴出某浏览器海外版，也正好是2年多前和团队战友们一起创造出的国际版本。

两个产品也都取得了很不错的海外市场成绩，实事求是的说，与国内产品的差别和适应国际用户的体验为成绩创造了客观条件，UC印度拿下头把交椅，hao123多个国家浏览器首页市场份额No.1。

说下产品与国内差异化过程中的考虑吧：

1、什么样的国际产品适合差异化？
中国出海产品现在越来越多，除了自己负责过的UC和Hao123，其它成功的，不成功的都很多，list下：从最早的网秦，易查，到后面的UC，金山cleanmaster，点心，再到微信，Go系列，各种游戏出海，所有公司都会面临一个问题：产品国际化时，是单发版，还是简单翻译？
基于成本和体验双重标准，很容易形成一个套路：

  • A 纯工具类产品、社交类产品：一套产品架构走天下，如点心。资源最小化，价值最大化。
  • B 涉及内容和本地化的产品：单独拉出一条支线，浏览器因为有导航及很多在线业务，因此，在UI层以上，也会单独出来。注意：更多的是UI层，并不是核心的框架层、内核层，数据协议、基础架构、后台上，很多产品国内、国际都依然是一套体系。

特别说一下，很多人会问：很多美国公司为啥都是一个UI版本？事实上也是基本满足A和B的，一些内容非常深度的网站，重点国家也会不同程度的差异化，如Amazon、Ebay、Yahoo，大家可以看下雅虎日本和雅虎美国，连主色系都快换了。

2、组织架构如何保证差异化落地？
在B的情况下，比较多的情况是，除了品牌名保持一致外，其他所有都可以推翻，老板希望新的团队基于国际市场发挥创造力。于是进一步，很多公司有了国际事业部，而国际部门为了追求自身的绝对独立和不断强化各线资源对国际业务的专业性，往往会组建隶属于国际业务的PM\UI\UE资源，有的甚至还有专属的研发资源。

3、为什么国际产品往往UI比国内好看？
一句话：国内是大象难转身，国际是轻装上阵。
1）国内业务，其受到的限制、约束不胜枚举，一个导航每时每刻正在创造的商业价值就决定了很多事情不能变动，一个产品团队需要考虑的太多，已经远远不只是UI、UE上的问题。
2）国内用户已经习惯的东西，不能简单以好看与不好看来衡量是否改动。淘宝做过简化尝试，但效果马上就跌了，因为用户当对一个产品某方面属性形成非常强的依赖时（比如导航网站的站点枚举，购物网站密码但固定的分类罗列），UI就会下降到一个相对其次的地位。产品经理更需要在这个需求方向上去延伸，去结合新的视觉设计，而不是为了美观而美观。
3）当国际业务无用户基础，可以从0切入时，想象空间于是就大了，所谓的后发优势。

4、最重要，国际产品都做什么样的差异化？
即便有单独的国际团队，也不能因为有资源而为了差异而差异。需搞清楚做什么差异化，做他的目的是什么？以Hao123为例，几乎每一个差异化都是基于国际市场的需求切合的：

  • 比如站点图像化，国际文字不像中文，95%的站点能用2个字能表达，于是图像化比文字化更能识别（这是国内、国际产品内容型产品非常重要的区别原因）
  • 比如分类区竖版，而非国内横版，因为多语言时，竖版能统一一套架构做到不同语种的规则排版。
  • 比如Hao123的开放平台，国际产品无法直接获取那么多的本地内容，开放平台是最好的接入本土开发者的产品架构
  • 比如左侧工具类，海外用户因为本土网站查询结果类步长过长，对快捷查询有非常大的需求。
    以上，即是差异化，也都是创新，都是基于真正用户需求的体验表达。

  • AWS介绍(1)

一、云计算的普及是IT行业必然趋势，很多企业在技术选型上都会考虑是公有云还是自建IDC机房，随着云服务的普及及成熟，公有云的方案不失为一种非常好的解决方法。根据公司的量级、安全性、成本等诸多方面考虑，云方案很适合中小企业，尤其创业企业，具备灵活、可扩展、快速等优势。基于IaaS公有云、私有云方案国内外都有很成熟的产品，比如国内的阿里云、腾讯云等、又拍云、七牛云存储、openstack等等，今天咱们介绍亚马逊AWS 。AWS来到中国应该是在2013年，目前AWS在全球有11个Region ，中国区有北京（2AZ，AZ指的是可用区）、宁夏两个Region ,限于中国互联网环境，中国区和全球服务是分离的，采用两套账户体系。而且有部分服务在中国区是没有提供的，例如cdn 。目前最接近大陆的CDN服务是通过边缘节点香港转过来的。AWS在新加坡、日本、悉尼，美国、欧洲的法拉克福、爱尔兰、南美洲的巴西都有Region 。当然AWS不仅仅是IAAS，也提供Paas服务 。

二、AWS提供IAM、EC2虚机 、ELB负载均衡、存储服务（对象、数据库、文档、NoSql）、数据仓库、镜像、VPC、IP漂移等等,AWS在技术上无疑是领先于阿里云等其他云服务的，在中国有些水土不服的地方，如域名备案、财务结算。目前中国区AWS只对部分企业开放并不针对个人。

三、责任公担模型
AWS负责：计算、存储、数据库、网络、可用区、边缘节点等
客户负责：平台、应用程序、操作系统、网络、防火墙
这样清晰的界定AWS的职责、AWS以更开放、包容的服务赢得公司信赖。

  • 谈谈创业公司的产品经理

一个码农谈产品经理，纯属扯马篮子，无奈很多创业公司都会出现的一种现象，产品经理对产品方向感不能很好把握，需求杂乱无章没有节制，程序瑗充当试验品，无休止增加需求，产品没有重点，做一个大而全，还很牛逼的说：我们是最全的，有个毛用。百度指数、CNZZ指标总觉得不够用，还要自己分析日志，你特马的已有指标看明白了么？ 定需求的时候，先拍脑门，说这事就这么定了。然后拍胸脯，说出了问题我负责，最后拍大腿，说我擦，当是谁开发的。其实这事和开发同学有个猫逼关系。产品经理拔吊无情啊，扪心自问你是合格的产品经理么，那么怎样成为合格的产品经理？ 身为程序猿的我，学习了馒头学院的课程 ，深受启发。产品经理把握整个产品的方向，公司的定位，应该从整个行业从CEO角度看整个行业形态；

产品经理四步工作法
❖ 找到痛点，定义场景
❖ 体验做透，方案优雅
❖ 保持克制，体验闭环
❖ 小步快跑，快速迭代

• 一个明确而且用户迫切需要的产品，很容易找到明确的用户群。
这样，产品研发出来后，不容易走偏。
• 选择的用户需求要有一定的普遍性，这点决定这个产品的未来市
场前景。
• 解决的问题少，开发速度快，也容易控制初期的研发成本和风险。
• 解决明确问题的产品，容易给用户说清楚，推广也会相对简单。

好的和普通的产品经理的区别？
• 好的的产品经理会说“嗯，让我们来做一个东西，它可
以帮助用户解决一个XX问题。”
• 普通产品经理会说“哈！这个东西太烂了，我能找出一
百个优化点！想出了一百个新功能！提升了一千项用户
体验！”

为什么小团队反而容易成功？
• 沟通成本低，95%的问题出在沟通上
• 因为资源有限，反而会专注在核心功能上
• 80%，40%原则 80%80%80%*80%=40%

传统行业微信商业化之道？
微信是一个快速吸纳粉丝的阵地，传统行业限于成本考虑不适合直接开发APP，请尝试微信营销，直接用NODEJS对接微信开放平台。

低频”互联网产品如何营销？
要综合客单价和消费频次两个因素。
客单价和利润率不好改变，那么就要从消费频次上做文章，提升用户粘性
产品分为：可运营产品，核心业务要运营的产品、运营产品 ，辅助运营的产品，如果你的核心产品是一个工具，比如IP地址查询，没有情感，那么就需要部署运营型产品提高情感挖掘可预期需求，让他有调动用户使用的空间。

运营的目的？
1、获客：获得你想要的用户
2、教育：让用户认识和使用你的服务
3、粘性：让用户多次使用你的服务

低频消费的获客方法？
竞争入口，获取精准用户 。购买流量是获取客户有效方式。适合足够有钱，产品相比竞争对手足够好的企业，用户对产品最强烈感知是价格和服务。
扩大范围，可以获取潜在客户，比如微信号、会员制，找到潜在用户，并能低成本获取他们，跨界沟通，分析和你行业相似行业，通过活动实现行业共享，以旧带新，用老用户拉新用户 ，拉身边朋友进来，口碑营销，或活动机制 。
